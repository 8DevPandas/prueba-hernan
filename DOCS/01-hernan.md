Qué es un sprint?

Un sprint es un periodo de tiempo fijo (generalmente de 1 a 4 semanas) en el que un equipo de desarrollo trabaja para completar un conjunto definido de tareas. Es la unidad de trabajo fundamental en la metodología Scrum.

## Cómo crear un sprint

### 1. Definir el objetivo del sprint (Sprint Goal)

Antes de comenzar, el equipo debe establecer un objetivo claro y conciso que responda a la pregunta: **¿Qué queremos lograr al final de este sprint?**

El objetivo debe ser:

- **Específico**: describir claramente qué se va a entregar.
- **Alcanzable**: realista para el tiempo disponible.
- **Medible**: que se pueda verificar si se cumplió o no.

Ejemplo: _"Implementar el flujo completo de registro de usuarios con validación de email."_

### 2. Realizar la planificación del sprint (Sprint Planning)

La planificación es una reunión donde participa todo el equipo. Se divide en dos partes:

**Parte 1 — Qué se va a hacer:**

- El Product Owner presenta los elementos del Product Backlog con mayor prioridad.
- El equipo analiza y discute cada elemento para entender los requisitos.
- Se seleccionan los elementos que se comprometen a completar durante el sprint.

**Parte 2 — Cómo se va a hacer:**

- El equipo descompone cada elemento seleccionado en tareas técnicas concretas.
- Se estima el esfuerzo de cada tarea (usando puntos de historia, horas, tallas de camiseta, etc.).
- Se asignan responsables iniciales (aunque las tareas pueden reasignarse durante el sprint).

### 3. Crear el Sprint Backlog

El Sprint Backlog es la lista de tareas que el equipo se compromete a completar. Para crearlo:

1. **Seleccionar las historias de usuario** del Product Backlog que se incluirán en el sprint.
2. **Descomponer cada historia en subtareas** técnicas (diseño, desarrollo, testing, etc.).
3. **Estimar cada subtarea** para asegurar que el total no exceda la capacidad del equipo.
4. **Priorizar las tareas** dentro del sprint para trabajar primero en lo más importante.
5. **Documentar los criterios de aceptación** de cada historia para saber cuándo está terminada.

### 4. Configurar el tablero del sprint

Organizar un tablero (físico o digital) con las siguientes columnas mínimas:

| Columna         | Descripción                                    |
| --------------- | ---------------------------------------------- |
| **Por hacer**   | Tareas que aún no se han comenzado             |
| **En progreso** | Tareas que alguien está trabajando activamente |
| **En revisión** | Tareas completadas esperando revisión o QA     |
| **Terminado**   | Tareas que cumplen los criterios de aceptación |

### 5. Establecer la duración y las fechas

- **Duración recomendada**: 2 semanas es lo más común, pero puede ser de 1 a 4 semanas.
- **Fecha de inicio**: generalmente el primer día hábil después del Sprint Planning.
- **Fecha de fin**: se fija al momento de crear el sprint y no se modifica.
- **Consistencia**: mantener la misma duración entre sprints para generar ritmo y previsibilidad.

### 6. Definir las ceremonias del sprint

Cada sprint incluye las siguientes reuniones:

- **Daily Standup** (diaria, 15 min): cada miembro responde qué hizo ayer, qué hará hoy y si tiene bloqueos.
- **Sprint Review** (al final del sprint): se demuestra el trabajo terminado a los stakeholders.
- **Sprint Retrospective** (al final del sprint): el equipo reflexiona sobre qué funcionó bien, qué se puede mejorar y qué acciones tomar.

### 7. Establecer la Definición de Terminado (Definition of Done)

Acordar criterios claros que toda tarea debe cumplir para considerarse terminada:

- El código está escrito y funciona correctamente.
- Se escribieron y pasaron las pruebas unitarias.
- Se realizó revisión de código (code review).
- La funcionalidad fue probada en un ambiente de staging.
- La documentación fue actualizada si corresponde.
- No hay bugs críticos abiertos relacionados.

### 8. Iniciar el sprint

Una vez completados los pasos anteriores:

1. Comunicar al equipo que el sprint ha comenzado oficialmente.
2. Asegurar que todos tienen acceso al tablero y al Sprint Backlog.
3. Comenzar a trabajar en las tareas priorizadas.
4. Realizar la Daily Standup desde el primer día.

## Buenas prácticas

- **No agregar trabajo nuevo** al sprint una vez iniciado, salvo excepciones críticas.
- **Proteger al equipo** de interrupciones externas durante el sprint.
- **Hacer visible el progreso** actualizando el tablero diariamente.
- **Revisar la velocidad del equipo** (cantidad de puntos completados por sprint) para mejorar las estimaciones futuras.
- **Celebrar los logros** al finalizar cada sprint para mantener la motivación del equipo.
